## 1. Architecture Overview

The application follows a **3-layer architecture** pattern:

```
┌─────────────────────────────────────────────────────────────────┐
│                        GUI LAYER (Tkinter)                      │
│  login_page → mfa_page → dashboard → add/view/delete/audit/exp │
├─────────────────────────────────────────────────────────────────┤
│                       SERVICES LAYER                            │
│  auth_service │ crypto_service │ password_service │ audit/export│
├─────────────────────────────────────────────────────────────────┤
│                       DATABASE LAYER                            │
│  connection.py (pool) │ queries.py (SQL constants)              │
├─────────────────────────────────────────────────────────────────┤
│                       UTILITIES                                 │
│  constants.py (.env loader) │ helpers.py (validation, logging)  │
└─────────────────────────────────────────────────────────────────┘
```

**Why 3 layers?**
- **Separation of concerns**: Each layer has one job. GUI handles display, services handle logic, DB handles data.
- **Testability**: You can test services without a GUI, and test DB queries without services.
- **Maintainability**: Changing the database doesn't affect the GUI, and vice versa.

### 3.3 Database Setup (setup_db.sql — detailed)

The SQL script performs 5 operations when run as MySQL root:

```
Step 1: CREATE DATABASE password_protect_db
         → Character set: utf8mb4 (supports emojis and international characters)
         → Collation: utf8mb4_unicode_ci (case-insensitive text comparison)

Step 2: CREATE USER 'admin'@'localhost'
         → Password: admin123**
         → @'localhost' means this user can ONLY connect from the same machine
           (no remote connections — security best practice)

Step 3: GRANT ALL PRIVILEGES ON password_protect_db.* TO 'admin'@'localhost'
         → The user has full access to this ONE database only
         → Cannot access any other databases on the MySQL server
         → FLUSH PRIVILEGES reloads the grant tables so changes take effect

Step 4: CREATE TABLE admin_users
         → Stores the admin account (username, bcrypt password hash, TOTP secret)

Step 5: CREATE TABLE passwords
         → Stores encrypted credentials (site, username, Fernet-encrypted password)

Step 6: CREATE TABLE audit_log
         → Records all actions (login, CRUD, export) with timestamps
```

The script is idempotent (safe to run multiple times) due to `IF NOT EXISTS` on all statements.

**Running on Windows:**
```bash
# If MySQL is not running:
net start mysql81

# Run the script:
mysql -u root -p < setup_db.sql
```

**Running on macOS:**
```bash
# If MySQL is not running:
brew services start mysql

# Run the script:
mysql -u root -p < setup_db.sql
```

### 4.2 Connection Pool (connection.py)

```
Application starts
    → get_pool() creates a pool of 5 MySQL connections (Singleton)
    → execute_query() borrows a connection, runs SQL, returns it
    → Connection is reused, not destroyed (efficient)
```

**Key design decisions:**
- **Singleton pattern**: Only one pool is ever created, no matter how many modules import it.
- **Parameterized queries**: All SQL uses `%s` placeholders to prevent SQL injection.
- **dictionary=True**: Results are returned as dicts (`{"id": 1, "site": "Gmail"}`) instead of tuples (`(1, "Gmail")`), making code more readable.
- **try/finally**: Connections are ALWAYS returned to the pool, even if an error occurs.

### 4.3 Query Constants (queries.py)

All SQL is centralized in one file as named string constants. This means:
- No SQL scattered across service files
- Easy to find and modify any query
- Consistent naming convention: `GET_*`, `INSERT_*`, `SELECT_*`, `COUNT_*`, `DELETE_*`

## 5. Services Layer

### 5.1 Crypto Service (crypto_service.py)

**Encryption algorithm**: Fernet (AES-128-CBC + HMAC-SHA256)

```
Admin password: "Admin1!"
        ↓
PBKDF2(password, salt, 480000 iterations) → 32-byte key
        ↓
Fernet(key) → encryption engine
        ↓
encrypt("MyP@ss") → "gAAAAABl..." (stored in DB)
decrypt("gAAAAABl...") → "MyP@ss" (shown in UI)
```

**Critical files:**
- `.salt` — Random 16-byte salt generated on first run. If deleted, ALL encrypted passwords become unrecoverable.

**Key functions:**
| Function | Purpose |
|----------|---------|
| `init_crypto(password)` | Derive key from admin password, initialize Fernet |
| `encrypt(plaintext)` | Encrypt a password for DB storage |
| `decrypt(ciphertext)` | Decrypt a password for display |
| `encrypt_for_export(data, pwd)` | Encrypt export data with a separate password |

### 5.2 Auth Service (auth_service.py)

**Two-step authentication:**

| Step | Method | Library |
|------|--------|---------|
| 1. Password | bcrypt hash comparison | `bcrypt` |
| 2. MFA | TOTP 6-digit code verification | `pyotp` |

**bcrypt explained:**
- One-way hash: password → hash (cannot reverse)
- Includes random salt: same password → different hash each time
- Intentionally slow: prevents brute-force attacks
- `checkpw()` re-hashes input and compares to stored hash

**TOTP (Time-based One-Time Password) — Deep Dive:**

TOTP is the industry-standard algorithm used by Google Authenticator, Microsoft
Authenticator, and similar apps. Here's how it works under the hood:

```
Both sides have:  secret = "JBSWY3DPEHPK3PXP"  (shared during setup)
Both sides know:  current_time = 1736956800      (Unix timestamp)

Code = HMAC-SHA1(secret, floor(current_time / 30))
     = HMAC-SHA1("JBSWY3DP...", 57898560)
     = ... truncated to 6 digits ...
     = 482901

The code changes every 30 seconds because floor(time / 30) changes.
```

**Key points:**
- The server NEVER contacts the phone. Both sides independently compute the
  same code because they share the same secret and use the same time.
- `valid_window=1` means the server accepts the current code AND the previous
  30-second code. This handles slight clock differences between the server and
  the phone (e.g., if the phone's clock is 10 seconds behind).
- The secret is a random Base32 string generated once during admin setup and
  stored in the `admin_users.totp_secret` column.

**TOTP in this application — two places:**

| Where | File | What Happens |
|-------|------|--------------|
| Setup (one-time) | `create_admin.py` | Generate secret → QR code → user scans → verify code |
| Login (every time) | `mfa_page.py` → `auth_service.verify_totp()` | User enters 6-digit code → server verifies |

### 5.3 Password Service (password_service.py)

**CRUD operations with pagination:**

| Function | SQL Used | Returns |
|----------|----------|---------|
| `add_password()` | INSERT_PASSWORD | New row ID |
| `get_passwords(page)` | SELECT_PASSWORDS + COUNT_PASSWORDS | (rows, total_pages) |
| `search_passwords(query, page)` | SEARCH_PASSWORDS + COUNT_SEARCH | (rows, total_pages) |
| `delete_password(id)` | GET_PASSWORD_BY_ID + DELETE_PASSWORD | True/False |
| `decrypt_password(encrypted)` | — (uses crypto_service) | Plain text |

**Pagination math:**
```python
PAGE_SIZE = 10
total_records = 25

page=1: OFFSET=0,  LIMIT=10 → rows 1-10
page=2: OFFSET=10, LIMIT=10 → rows 11-20
page=3: OFFSET=20, LIMIT=10 → rows 21-25

total_pages = ceil(25 / 10) = 3
```

| Format | Security | Use Case |
|--------|----------|----------|
| CSV | ⚠️ Plain text | Quick import to Excel/Sheets |
| JSON | ⚠️ Plain text | Programmatic processing |
| Encrypted (.enc) | ✅ Fernet encrypted | Secure backup/transfer |

**Decrypting exported .enc files:**

The app includes a standalone CLI script `decrypt_export.py` for decrypting
exported files. It uses the same PBKDF2 + Fernet logic as the main app.

```bash
uv run python decrypt_export.py
```

Requirements for decryption:
- The `.enc` file path
- The export password entered during export
- The `.salt` file from the project root (must be the same `.salt` that existed at export time)

**Full interactive session example:**

```
=== Password Protect — Decrypt Exported File ===

Path to .enc file: C:\Users\Documents\passwords_export.enc
Export password: ********

Decrypted 3 password(s):

[
  {
    "id": 1,
    "site_name": "Gmail",
    "username": "john@gmail.com",
    "password": "MyP@ssw0rd",
    "created_at": "2025-01-15 10:30:00"
  },
  ...
]

Save to file? (y/n): y
Output file path (e.g., decrypted.json): decrypted.json
Saved to decrypted.json
```

**How to provide the .enc file path at the prompt:**
- Type the full path directly: `C:\Users\Documents\export.enc`
- Or drag and drop the `.enc` file from File Explorer into the terminal window
  (the OS auto-pastes the full path)
- Paths with spaces work as-is — the script strips surrounding quotes automatically
- On macOS: `/Users/admin/Desktop/export.enc` (forward slashes)

**Error scenarios:**

| Error Message | Cause | Fix |
|---------------|-------|-----|
| `File not found` | Wrong path or typo | Check the path and try again |
| `.salt file not found` | Missing .salt at project root | Copy the .salt from the original machine |
| `Decryption failed. Wrong password or different .salt file` | Wrong export password OR different .salt | Use the exact password from export time + the original .salt file |

The script derives the same Fernet key using `PBKDF2(export_password + .salt)`,
decrypts the file, and outputs the JSON array of passwords. It optionally saves
the decrypted data to a file.

**Why a separate script?**
- The exported file should be decryptable independently of the running app
- Useful for disaster recovery (DB is down but you have the .enc backup)
- Can be run on any machine with Python + cryptography installed + the .salt file

## 6. GUI Layer (Tkinter)

### 6.1 Frame Stacking Pattern (main.py)

All 8 pages are created at startup and stacked in the same grid cell:

```python
# All frames occupy row=0, column=0 — stacked like cards
frame.grid(row=0, column=0, sticky="nsew")

# show_frame() brings a specific page to the front
frame.tkraise()  # Raises this frame above all others
```

**Why stack instead of create/destroy?**
- Faster navigation (no widget creation overhead)
- State is preserved (form fields keep their values)
- Simpler code (no dynamic widget management)

### 6.2 Controller Pattern

The `App` class is passed to every page as `controller`:

```python
# Page navigates to another page:
self.controller.show_frame("Dashboard")

# Page accesses the logged-in user:
admin = self.controller.current_user["username"]

# Page triggers logout:
self.controller.logout()
```

### 6.3 Page Lifecycle — on_show()

Every page can define an `on_show(**kwargs)` method. It's called automatically
by `show_frame()` before the page becomes visible. Used for:
- Resetting form fields (LoginPage clears username/password)
- Refreshing data (ViewPasswordsPage reloads from DB)
- Receiving parameters (MfaPage receives the user dict)

## 7. Security Design

### 7.1 Defense in Depth

| Layer | Protection |
|-------|-----------|
| Authentication | bcrypt password hashing + TOTP MFA |
| Encryption | Fernet (AES) with PBKDF2-derived key |
| Database | Parameterized queries prevent SQL injection |
| Configuration | Secrets in .env (gitignored), not in code |
| Audit | Every action logged to DB and file |
| Validation | All inputs validated before processing |
| DB User | MySQL user has access to ONE database only |

### 7.2 What Happens If...

| Scenario | Impact | Mitigation |
|----------|--------|-----------|
| Database stolen | Passwords are encrypted, unreadable | Fernet encryption |
| .env file leaked | DB credentials exposed | Use strong DB passwords in production |
| .salt file deleted | All encrypted passwords lost forever | Back up the .salt file |
| Brute-force login | Attacker tries many passwords | Audit log tracks LOGIN_FAILED events |
| SQL injection attempt | Attacker tries to manipulate queries | Parameterized queries (%s) prevent this |

### 7.3 Password Storage Comparison

```
❌ Plain text:     "MyP@ssw0rd"           → Anyone can read it
❌ Base64 encoded: "TXlQQHNzdzByZA=="     → Trivially reversible
❌ MD5 hash:       "5f4dcc3b5aa765d61..."  → Rainbow table attacks
✅ bcrypt hash:    "$2b$12$LJ3m5ZQn..."    → Slow, salted, secure (for login)
✅ Fernet encrypt: "gAAAAABl..."           → Reversible with key (for stored passwords)
```

### 8.2 MFA Setup Flow (create_admin.py — detailed)

This flow runs ONCE when the admin user is first created. It establishes the
shared secret between the server and the authenticator app.

```
┌──────────────────────────────────────────────────────────────────────────┐
│  STEP 1: Generate Secret                                                │
│  pyotp.random_base32() → "JBSWY3DPEHPK3PXP"                           │
│  A random 32-character Base32 string. This is the SHARED SECRET that   │
│  both the server and the authenticator app will use to generate codes.  │
├──────────────────────────────────────────────────────────────────────────┤
│  STEP 2: Create Provisioning URI                                        │
│  pyotp.TOTP(secret).provisioning_uri(name=username, issuer="...")      │
│  → "otpauth://totp/PasswordProtect:admin?secret=JBSWY3DP&issuer=..."  │
│  This URI follows the otpauth:// protocol that all authenticator apps  │
│  understand. It tells the app: add account "admin" for "PasswordProtect"│
│  using this secret key.                                                 │
├──────────────────────────────────────────────────────────────────────────┤
│  STEP 3: Display QR Code in Terminal                                    │
│  qrcode.QRCode(box_size=1, border=1)                                   │
│  qr.add_data(uri) → qr.make() → qr.print_ascii(invert=True)          │
│  The provisioning URI is encoded into an ASCII QR code printed in the  │
│  terminal. The admin opens their authenticator app, taps "Add account" │
│  → "Scan QR code", and points the camera at the terminal.             │
│  The app now has the shared secret.                                     │
├──────────────────────────────────────────────────────────────────────────┤
│  STEP 4: Verify Setup (Proof of Configuration)                          │
│  The admin enters the 6-digit code currently showing on their phone.   │
│  pyotp.TOTP(secret).verify(code, valid_window=1)                       │
│  The server independently generates the expected code using the same   │
│  secret + current time, and checks if they match.                      │
│  This confirms the QR code was scanned correctly.                      │
├──────────────────────────────────────────────────────────────────────────┤
│  STEP 5: Save to Database                                               │
│  execute_query(INSERT_ADMIN, (username, password_hash, totp_secret))    │
│  The totp_secret is stored in admin_users.totp_secret column.          │
│  Later, mfa_page.py reads this secret to verify codes at login time.   │
└──────────────────────────────────────────────────────────────────────────┘
```

**MFA Login Flow (every subsequent login via mfa_page.py):**
```
1. Admin enters username + password on LoginPage → authenticated
2. LoginPage checks: is_mfa_enabled() AND user has totp_secret?
3. If yes → navigate to MfaPage, passing the user dict
4. MfaPage shows a 6-digit code input field
5. Admin opens authenticator app, reads current code, types it in
6. MfaPage calls auth_service.verify_totp(user["totp_secret"], code)
7. pyotp.TOTP(secret).verify(code, valid_window=1)
8. If match → audit "MFA_VERIFIED" → navigate to Dashboard
9. If no match → show error → audit "MFA_FAILED" → stay on page
```

### 8.3 Add Password Flow (detailed)

```
1. Admin fills form: site="Gmail", user="john", pass="MyP@ss1!"
2. _save() is called
3. validate_site_name("Gmail")     → OK
4. validate_username("john")       → OK
5. validate_password("MyP@ss1!")   → OK (meets all standards)
6. confirm == password             → OK
7. crypto_service.encrypt("MyP@ss1!") → "gAAAAABl..."
8. execute_query(INSERT_PASSWORD, ("Gmail", "john", "gAAAAABl..."))
9. audit_service.log_action("ADD_PASSWORD", "Added for Gmail", "admin")
10. Success message shown, form cleared
```

### 8.4 View + Reveal Flow (detailed)

```
1. on_show() calls _load()
2. get_passwords(page=1) → fetches 10 rows from DB
3. Each row displayed with "••••••••" as password
4. Encrypted password stored in Treeview tags (hidden metadata)
5. Admin selects row, clicks "Reveal"
6. _reveal() reads tags[0] → "gAAAAABl..."
7. crypto_service.decrypt("gAAAAABl...") → "MyP@ss1!"
8. Table cell updated: "••••••••" → "MyP@ss1!"
9. Audit logged: VIEW_PASSWORD
```

## 10. Code Quality — Linting & Formatting (PEP 8 + Ruff)

### 10.1 What is PEP 8?

PEP 8 is the official Python style guide. It defines conventions for:
- Indentation (4 spaces, no tabs)
- Line length (max 120 characters in this project)
- Import ordering (stdlib → third-party → local)
- Naming conventions (snake_case for functions/variables, PascalCase for classes)
- Whitespace rules (spaces around operators, after commas, etc.)

Following PEP 8 makes code consistent and readable across the entire team.

### 10.2 What is Ruff?

Ruff is an extremely fast Python linter and formatter written in Rust.
It replaces multiple tools (flake8, isort, pycodestyle, bandit) with a single tool.
It's configured in `pyproject.toml` under `[tool.ruff]`.

### 10.3 Ruff Rules Enabled in This Project

| Rule Set | Code | What It Checks |
|----------|------|----------------|
| Pyflakes | `F` | Unused imports, undefined names, redefined variables |
| pycodestyle errors | `E` | PEP 8 style violations (indentation, whitespace, etc.) |
| pycodestyle warnings | `W` | PEP 8 style warnings (trailing whitespace, blank lines) |
| isort | `I` | Import sorting and grouping (stdlib → third-party → local) |
| pep8-naming | `N` | Naming conventions (snake_case functions, PascalCase classes) |
| pyupgrade | `UP` | Suggests modern Python 3.11+ syntax |
| flake8-bugbear | `B` | Common bug patterns (mutable default args, assert usage) |
| flake8-bandit | `S` | Security checks (hardcoded passwords, SQL injection risks) |
| flake8-comprehensions | `C4` | Simplify list/dict/set comprehensions |
| flake8-simplify | `SIM` | Simplify code patterns (unnecessary else, nested ifs) |

### 10.4 Rules Intentionally Ignored

| Rule | Why Ignored |
|------|-------------|
| `E501` (line too long) | Handled by the ruff formatter instead |
| `S311` (pseudo-random) | Not relevant — we use `os.urandom()` for crypto |
| `S105` in queries.py | False positive — variable names contain "PASSWORD" but values are SQL, not secrets |
| `S106` in create_admin.py | False positive — it's a setup script, not hardcoded credentials |
| `B008` (function call in default arg) | Used intentionally in Tkinter widget construction |

### 10.5 Import Ordering Convention

Ruff's isort rules enforce this import order in every file:

```python
# 1. Standard library imports (built into Python)
import math
import os

# 2. Third-party imports (installed via uv/pip)
import bcrypt
import pyotp

# 3. Local application imports (our own code)
from app.db.connection import execute_query
from app.utils.helpers import setup_logger
```

A blank line separates each group. Within each group, imports are alphabetically sorted.

### 10.6 Commands

```bash
# Check for lint issues (does NOT modify files)
uv run ruff check .

# Auto-fix all fixable lint issues (modifies files)
uv run ruff check --fix .

# Format all files (consistent style — modifies files)
uv run ruff format .

# Verify formatting without changing files (useful in CI/CD)
uv run ruff format --check .
```

### 10.7 Ruff Configuration Location

All ruff settings live in `pyproject.toml` under these sections:
- `[tool.ruff]` — Base settings (target version, line length, excludes)
- `[tool.ruff.lint]` — Which rules to enable/ignore
- `[tool.ruff.lint.per-file-ignores]` — File-specific rule overrides
- `[tool.ruff.lint.isort]` — Import sorting configuration
- `[tool.ruff.format]` — Formatter settings (quote style, indent style)

## 11. Troubleshooting

| Problem | Cause | Solution |
|---------|-------|----------|
| "Access denied for user 'admin'" | MySQL user not created | Run `mysql -u root -p < setup_db.sql` |
| "Crypto not initialized" | Logged in without init_crypto | This shouldn't happen — check login_page.py |
| "InvalidToken" on decrypt | Wrong password or .salt changed | Ensure .salt file hasn't been modified |
| All encrypted passwords lost | .salt file deleted | Restore from backup (no other recovery) |
| "No module named 'app'" | Running from wrong directory | Run from project root: `uv run python main.py` |
| MFA code always invalid | Clock skew > 60 seconds | Sync your computer's clock |
| Blank table on View page | No passwords added yet | Add passwords first via Add Password page |
| "Can't connect to MySQL" | MySQL service not running | Start MySQL: `net start mysql` (Windows) or `brew services start mysql` (macOS) |
